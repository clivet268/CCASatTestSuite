 13                  HyStart++: Modified Slow Start for TCP
 14 
 15 Abstract
 16 
 17    This document describes HyStart++, a simple modification to the slow
 18    start phase of congestion control algorithms.  Slow start can
 19    overshoot the ideal send rate in many cases, causing high packet loss
 20    and poor performance.  HyStart++ uses increase in round-trip delay as
 21    a heuristic to find an exit point before possible overshoot.  It also
 22    adds a mitigation to prevent jitter from causing premature slow start
 23    exit.

 72 1.  Introduction
 73 
 74    [RFC5681] describes the slow start congestion control algorithm for
 75    TCP.  The slow start algorithm is used when the congestion window
 76    (cwnd) is less than the slow start threshold (ssthresh).  During slow
 77    start, in the absence of packet loss signals, TCP increases the cwnd
 78    exponentially to probe the network capacity.  This fast growth can
 79    overshoot the ideal sending rate and cause significant packet loss
 80    that cannot always be recovered efficiently.
 81 
 82    HyStart++ builds upon Hybrid Start (HyStart), originally described in
 83    [HyStart].  HyStart++ uses increase in round-trip delay as a signal
 84    to exit slow start before potential packet loss occurs as a result of
 85    overshoot.  This is one of two algorithms specified in [HyStart] for
 86    finding a safe exit point for slow start.  After the slow start exit,
 87    a new Conservative Slow Start (CSS) phase is used to determine
 88    whether the slow start exit was premature and to resume slow start.
 89    This mitigation improves performance in the presence of jitter.
 90    HyStart++ reduces packet loss and retransmissions, and improves
 91    goodput in lab measurements and real-world deployments.
 92 
 93    While this document describes HyStart++ for TCP, it can also be used
 94    for other transport protocols that use slow start, such as QUIC
 95    [RFC9002] or the Stream Control Transmission Protocol (SCTP)
 96    [RFC9260].

106 3.  Definitions
107 
108    To aid the reader, we repeat some definitions from [RFC5681]:
109 
110    SENDER MAXIMUM SEGMENT SIZE (SMSS):  The size of the largest segment
111       that the sender can transmit.  This value can be based on the
112       maximum transmission unit of the network, the Path MTU Discovery
113       algorithm [RFC1191] [RFC4821], RMSS (see next item), or other
114       factors.  The size does not include the TCP/IP headers and
115       options.
116 
117    RECEIVER MAXIMUM SEGMENT SIZE (RMSS):  The size of the largest
118       segment that the receiver is willing to accept.  This is the value
119       specified in the MSS option sent by the receiver during connection
120       startup.  Or, if the MSS option is not used, it is 536 bytes
121       [RFC1122].  The size does not include the TCP/IP headers and
122       options.
123 
124    RECEIVER WINDOW (rwnd):  The most recently advertised receiver
125       window.
126 
127    CONGESTION WINDOW (cwnd):  A TCP state variable that limits the
128       amount of data a TCP can send.  At any given time, a TCP MUST NOT
129       send data with a sequence number higher than the sum of the
130       highest acknowledged sequence number and the minimum of the cwnd
131       and rwnd.
132 
133 4.  HyStart++ Algorithm
134 
135 4.1.  Summary
136 
137    [HyStart] specifies two algorithms (a "Delay Increase" algorithm and
138    an "Inter-Packet Arrival" algorithm) to be run in parallel to detect
139    that the sending rate has reached capacity.  In practice, the Inter-
140    Packet Arrival algorithm does not perform well and is not able to
141    detect congestion early, primarily due to ACK compression.  The idea
142    of the Delay Increase algorithm is to look for spikes in RTT (round-
143    trip time), which suggest that the bottleneck buffer is filling up.
144 
145    In HyStart++, a TCP sender uses standard slow start and then uses the
146    Delay Increase algorithm to trigger an exit from slow start.  But
147    instead of going straight from slow start to congestion avoidance,
148    the sender spends a number of RTTs in a Conservative Slow Start (CSS)
149    phase to determine whether the exit from slow start was premature.
150    During CSS, the congestion window is grown exponentially in a fashion
151    similar to regular slow start, but with a smaller exponential base,
152    resulting in less aggressive growth.  If the RTT reduces during CSS,
153    it's concluded that the RTT spike was not related to congestion
154    caused by the connection sending at a rate greater than the ideal
155    send rate, and the connection resumes slow start.  If the RTT
156    inflation persists throughout CSS, the connection enters congestion
157    avoidance.
158 
159 4.2.  Algorithm Details
160 
161    The following pseudocode uses a limit, L, to control the
162    aggressiveness of the cwnd increase during both standard slow start
163    and CSS.  While an arriving ACK may newly acknowledge an arbitrary
164    number of bytes, the HyStart++ algorithm limits the number of those
165    bytes applied to increase the cwnd to L*SMSS bytes.
166 
167    lastRoundMinRTT and currentRoundMinRTT are initialized to infinity at
168    the initialization time.  currRTT is the RTT sampled from the latest
169    incoming ACK and initialized to infinity.
170 
171    lastRoundMinRTT = infinity
172    currentRoundMinRTT = infinity
173    currRTT = infinity
174 
175    HyStart++ measures rounds using sequence numbers, as follows:
176 
177    *  Define windowEnd as a sequence number initialized to SND.NXT.
178 
179    *  When windowEnd is ACKed, the current round ends and windowEnd is
180       set to SND.NXT.
181 
182    At the start of each round during standard slow start [RFC5681] and
183    CSS, initialize the variables used to compute the last round's and
184    current round's minimum RTT:
185 
186    lastRoundMinRTT = currentRoundMinRTT
187    currentRoundMinRTT = infinity
188    rttSampleCount = 0
189 
190    For each arriving ACK in slow start, where N is the number of
191    previously unacknowledged bytes acknowledged in the arriving ACK:
192 
193    Update the cwnd:
194 
195      cwnd = cwnd + min(N, L * SMSS)
196 
197    Keep track of the minimum observed RTT:
198 
199      currentRoundMinRTT = min(currentRoundMinRTT, currRTT)
200      rttSampleCount += 1
201 
202    For rounds where at least N_RTT_SAMPLE RTT samples have been obtained
203    and currentRoundMinRTT and lastRoundMinRTT are valid, check to see if
204    delay increase triggers slow start exit:
205 
206    if ((rttSampleCount >= N_RTT_SAMPLE) AND
207        (currentRoundMinRTT != infinity) AND
208        (lastRoundMinRTT != infinity))
209      RttThresh = max(MIN_RTT_THRESH,
210        min(lastRoundMinRTT / MIN_RTT_DIVISOR, MAX_RTT_THRESH))
211      if (currentRoundMinRTT >= (lastRoundMinRTT + RttThresh))
212        cssBaselineMinRtt = currentRoundMinRTT
213        exit slow start and enter CSS
214 
215    For each arriving ACK in CSS, where N is the number of previously
216    unacknowledged bytes acknowledged in the arriving ACK:
217 
218    Update the cwnd:
219 
220    cwnd = cwnd + (min(N, L * SMSS) / CSS_GROWTH_DIVISOR)
221 
222    Keep track of the minimum observed RTT:
223 
224    currentRoundMinRTT = min(currentRoundMinRTT, currRTT)
225    rttSampleCount += 1
226 
227    For CSS rounds where at least N_RTT_SAMPLE RTT samples have been
228    obtained, check to see if the current round's minRTT drops below
229    baseline (cssBaselineMinRtt) indicating that slow start exit was
230    spurious:
231 
232    if (currentRoundMinRTT < cssBaselineMinRtt)
233      cssBaselineMinRtt = infinity
234      resume slow start including HyStart++
235 
236    CSS lasts at most CSS_ROUNDS rounds.  If the transition into CSS
237    happens in the middle of a round, that partial round counts towards
238    the limit.
239 
240    If CSS_ROUNDS rounds are complete, enter congestion avoidance by
241    setting the ssthresh to the current cwnd.
242 
243    ssthresh = cwnd
244 
245    If loss or Explicit Congestion Notification (ECN) marking is observed
246    at any time during standard slow start or CSS, enter congestion
247    avoidance by setting the ssthresh to the current cwnd.
248 
249    ssthresh = cwnd

251 4.3.  Tuning Constants and Other Considerations
252 
253    It is RECOMMENDED that a HyStart++ implementation use the following
254    constants:
255 
256    MIN_RTT_THRESH = 4 msec
257    MAX_RTT_THRESH = 16 msec
258    MIN_RTT_DIVISOR = 8
259    N_RTT_SAMPLE = 8
260    CSS_GROWTH_DIVISOR = 4
261    CSS_ROUNDS = 5
262    L = infinity if paced, L = 8 if non-paced
263 
264    These constants have been determined with lab measurements and real-
265    world deployments.  An implementation MAY tune them for different
266    network characteristics.
267 
268    The delay increase sensitivity is determined by MIN_RTT_THRESH and
269    MAX_RTT_THRESH.  Smaller values of MIN_RTT_THRESH may cause spurious
270    exits from slow start.  Larger values of MAX_RTT_THRESH may result in
271    slow start not exiting until loss is encountered for connections on
272    large RTT paths.
273 
274    MIN_RTT_DIVISOR is a fraction of RTT to compute the delay threshold.
275    A smaller value would mean a larger threshold and thus less
276    sensitivity to delay increase, and vice versa.
277 
278    While all TCP implementations are REQUIRED to take at least one RTT
279    sample each round, implementations of HyStart++ are RECOMMENDED to
280    take at least N_RTT_SAMPLE RTT samples.  Using lower values of
281    N_RTT_SAMPLE will lower the accuracy of the measured RTT for the
282    round; higher values will improve accuracy at the cost of more
283    processing.
284 
285    The minimum value of CSS_GROWTH_DIVISOR MUST be at least 2.  A value
286    of 1 results in the same aggressive behavior as regular slow start.
287    Values larger than 4 will cause the algorithm to be less aggressive
288    and maybe less performant.
289 
290    Smaller values of CSS_ROUNDS may miss detecting jitter, and larger
291    values may limit performance.
292 
293    Packet pacing [ASA00] is a possible mechanism to avoid large bursts
294    and their associated harm.  A paced TCP implementation SHOULD use L =
295    infinity.  Burst concerns are mitigated by pacing, and this setting
296    allows for optimal cwnd growth on modern networks.
297 
298    For TCP implementations that pace to mitigate burst concerns, L
299    values smaller than infinity may suffer performance problems due to
300    slow cwnd growth in high-speed networks.  For non-paced TCP
301    implementations, L values smaller than 8 may suffer performance
302    problems due to slow cwnd growth in high-speed networks; L values
303    larger than 8 may cause an increase in burstiness and thereby loss
304    rates, and result in poor performance.
305 
306    An implementation SHOULD use HyStart++ only for the initial slow
307    start (when the ssthresh is at its initial value of arbitrarily high
308    per [RFC5681]) and fall back to using standard slow start for the
309    remainder of the connection lifetime.  This is acceptable because
310    subsequent slow starts will use the discovered ssthresh value to exit
311    slow start and avoid the overshoot problem.  An implementation MAY
312    use HyStart++ to grow the restart window [RFC5681] after a long idle
313    period.
314 
315    In application-limited scenarios, the amount of data in flight could
316    fall below the bandwidth-delay product (BDP) and result in smaller
317    RTT samples, which can trigger an exit back to slow start.  It is
318    expected that a connection might oscillate between CSS and slow start
319    in such scenarios.  But this behavior will neither result in a
320    connection prematurely entering congestion avoidance nor cause
321    overshooting compared to slow start.

323 5.  Deployments and Performance Evaluations
324 
325    At the time of this writing, HyStart++ as described in this document
326    has been default enabled for all TCP connections in the Windows
327    operating system for over two years with pacing disabled and an
328    actual L = 8.
329 
330    In lab measurements with Windows TCP, HyStart++ shows goodput
331    improvements as well as reductions in packet loss and retransmissions
332    compared to standard slow start.  For example, across a variety of
333    tests on a 100 Mbps link with a bottleneck buffer size of bandwidth-
334    delay product, HyStart++ reduces bytes retransmitted by 50% and
335    retransmission timeouts (RTOs) by 36%.
336 
337    In an A/B test where we compared an implementation of HyStart++
338    (based on an earlier draft version of this document) to standard slow
339    start across a large Windows device population, out of 52 billion TCP
340    connections, 0.7% of connections move from 1 RTO to 0 RTOs and
341    another 0.7% of connections move from 2 RTOs to 1 RTO with HyStart++.
342    This test did not focus on send-heavy connections, and the impact on
343    send-heavy connections is likely much higher.  We plan to conduct
344    more such production experiments to gather more data in the future.
345 
346 6.  Security Considerations
347 
348    HyStart++ enhances slow start and inherits the general security
349    considerations discussed in [RFC5681].
350 
351    An attacker can cause HyStart++ to exit slow start prematurely and
352    impair the performance of a TCP connection by, for example, dropping
353    data packets or their acknowledgments.
354 
355    The ACK division attack outlined in [SCWA99] does not affect
356    HyStart++ because the congestion window increase in HyStart++ is
357    based on the number of bytes newly acknowledged in each arriving ACK
358    rather than by a particular constant on each arriving ACK.
